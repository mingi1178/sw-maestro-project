Fairytale · AI Module

일기 → 동화 생성 Agentic Workflow

LangGraph 위에서 3개의 LLM 에이전트(기획자 · 작가 · 편집자)가 협업해서 한 편의 동화를 만든다. 각 단계는 정해진 자율성을 갖고 결정을 내리며, 미달 시 코드 레벨의 안전장치가 개입한다. 아래 문서는 backend/app/graph/nodes.py의 실제 동작을 설명한다.

정상 진행 재집필 루프 (조건부) END 분기

정상 케이스의 LLM 호출은 3회(plan + write + critique). 첫 critique 미달 시 1회 재집필이 들어가 최대 5회(plan + write×2 + critique×2)에서 강제 종료된다.

에이전트의 자율성과 안전장치

각 노드는 LLM이 자유롭게 결정하는 부분과 코드가 강제하는 부분이 구분되어 있다. 자율성은 창의성을 위해 허용하되, 미달·파싱 실패·안전 위반은 코드 레벨에서 차단한다.

노드 LLM 자율성 (스스로 결정) 코드 강제 (자율성 제약)
PLAN setting/protagonist/마법 장치/갈등/결말의 구체 내용 전부. 어떤 동물·자연물로 의인화할지, 마법 장치의 모티프를 무엇으로 잡을지. JSON 스키마 6개 슬롯 강제 · 인물 카테고리별 매핑 규칙 (가족=보호자, 친구=갈등/협력, 선생님=안내자) · 파싱 실패 시 raw 텍스트 fallback.
WRITE 문체·운율·의성어/의태어 선택, 사건 묘사 디테일, 캐릭터 대사. 재집필 시 critique 코멘트를 어떻게 반영할지의 판단. 분량 6~10문장 · 5~7세 어휘 · 폭력/공포/혐오/직접 훈계 금지 · JSON 출력 형식 · 깨진 JSON은 정규식 폴백으로 복구.
CRITIQUE 4축 1~5점 채점 · 자유 코멘트로 어디를 어떻게 고치라고 지시 (이게 다음 write의 입력이 됨). 점수 1~5 클램핑 · 누락 항목 보정 · safety 누락은 0점 강제 · 파싱 실패 시 강제 미통과.
FINALIZE 없음 (LLM 호출 없음). 마지막 draft를 story로 단순 복사. 빈 body는 service 레이어에서 StoryParsingError.
STAGE ①

PLAN — 기획자 LLM

일기와 인물 정보를 동화 세계의 뼈대(JSON)로 변환한다. 본문은 아직 쓰지 않는다.

입력 / 출력

  • 입력: diary, people (가족/친구/선생님/이웃/기타로 그룹화된 텍스트)
  • 출력 슬롯: setting, protagonist, companions[], magical_element, conflict, resolution

자율성

"엄마"를 달빛 사슴으로 바꿀지 곰 할머니로 바꿀지, "블록 다툼"을 어떤 마법 장치 갈등으로 풀지는 LLM이 결정한다. 이 자유도는 같은 일기로도 매번 다른 동화가 나오게 만드는 핵심이다.

✅ 통과 조건

JSON 파싱 성공 → outline을 indent 2로 정돈해서 다음 노드로 전달.

❌ 미달 / 실패 시

  • JSON 파싱 실패: 코드펜스 제거 + {...} brace 추출로 재시도. 그래도 실패하면 raw 텍스트 그대로 outline에 저장 (write 노드가 텍스트로라도 읽을 수 있게).
  • LLM 호출 자체 실패: _safe_invoke가 LLMUnavailableError(stage="plan")로 감싸 throw → 그래프 중단 → service에서 503으로 매핑.
  • 스키마 슬롯이 빠져도 별도 검증은 없다 (write 단계의 프롬프트가 outline의 결함을 보완).
STAGE ②

WRITE — 작가 LLM

outline을 따라 한 편의 동화를 쓴다. 재호출 시에는 직전 critique 피드백을 반영해 고쳐 쓴다.

입력 / 출력

  • 입력: diary, people, outline, (재호출 시) critique.scores + critique.comment
  • 출력: { "title": "...", "body": "..." }

자율성

도입부 ("옛날 옛적", "어느 별빛 흐르는 밤에"...), 의성어/의태어 선택, 운율, 캐릭터 대사·묘사를 LLM이 결정한다. 특히 재집필 시 critique이 던진 자유 코멘트를 어떻게 반영할지의 판단이 작가의 핵심 자율성이다 — "톤이 너무 무겁다"는 지적을 보고 의성어를 추가할지, 결말 분위기를 바꿀지를 LLM이 스스로 정한다.

제약 (시스템 프롬프트)

  • 분량 6~10 문장, 5~7세 어휘
  • 일기의 현실 사물·장소를 outline의 마법 장치로 변환해서 사용
  • 폭력/공포/혐오/직접 훈계 금지 — 교훈은 결말 분위기로만

✅ 통과 조건

응답에서 title/body 추출 성공 → draft_title, draft_body, revision_count += 1 갱신.

❌ 미달 / 실패 시

  • JSON 깨짐: _parse_story가 3단계 폴백 — ① 정상 JSON 파싱 → ② 정규식 "title":"..." / "body":"..." 추출 → ③ 마지막 수단으로 title="오늘의 동화", body=원문 그대로.
  • 분량/어휘 위반: 코드는 직접 검증하지 않는다. 다음 단계의 critique이 채점으로 잡아내고 재집필을 트리거한다.
  • LLM 호출 실패: LLMUnavailableError(stage="write").
재집필 루프의 작동: revision_count > 0이면 user 프롬프트에 "[이전 원고에 대한 비평가의 지적 — 반드시 반영해서 다시 쓸 것]" 블록이 추가된다. LLM은 이전 자기 출력이 아니라 비평을 보고 다시 쓴다 — self-refine 패턴.
STAGE ③

CRITIQUE — 편집자 LLM

방금 쓰인 draft를 4축 1~5점으로 채점하고 자유 코멘트를 단다. 통과/실패 라우팅의 근거가 된다.

4개 채점 축 (CRITIQUE_KEYS)

키의미
fairy_tale동화적 분위기 (옛날옛적 톤, 마법 장치, 의인화, 의성어/의태어)
age_vocab5~7세 어휘 적절성
character_mapping입력 인물의 카테고리/역할이 캐릭터에 잘 녹아들었는가
safety공포·폭력·직접 훈계·차별 표현이 없는가

통과 기준

average ≥ PASS_THRESHOLD (4.0) AND 파싱 성공 → passed=True.

자율성

각 항목에 몇 점을 줄지, 어떤 부분을 어떻게 고치라고 코멘트할지는 전적으로 LLM이 결정한다. 이 코멘트가 다음 write의 프롬프트에 그대로 들어가므로 편집자 LLM은 작가 LLM에게 직접 지시를 내리는 권한을 갖는다.

✅ 통과 (passed=True)

route_after_critique가 "finalize" 반환 → 다음은 finalize.

❌ 미달 / 파싱 실패 시

  • 점수 클램핑: _clamp_score가 정수 변환 실패 시 3점, 1~5 범위 강제.
  • safety 누락: 0점으로 강제 — 평균이 절대 4.0을 못 넘게 한다 (위험 콘텐츠 차단).
  • 그 외 항목 누락: 보수적으로 3점.
  • JSON 파싱 자체 실패: parsing_failed=True + safety 0점 + passed=False. 위험 콘텐츠가 통과되는 경로 차단.
  • route_after_critique: passed=False이고 revision_count ≤ MAX_REVISIONS(=1)이면 "write" 반환 → 재집필 루프 진입.
핵심 안전장치: safety 누락 시 0점 강제는, "LLM이 safety 항목을 빼먹는 행위 = 평가 불가능 = 통과 불가"로 처리하는 보수적 정책이다. 무한 루프는 MAX_REVISIONS 한도로 막는다.

라우팅: route_after_critique

def route_after_critique(state):
    crit = state.get("critique") or {}
    rev = state.get("revision_count", 0)
    if crit.get("passed") or rev > MAX_REVISIONS:
        return "finalize"
    return "write"
상황passedrevision_count다음
1차 critique 통과True1finalize
1차 미달 → 재집필False1write (2차)
2차도 미달 → 한도 초과False2finalize 강제
2차 통과True2finalize

최악의 경우에도 LLM 호출은 plan(1) + write(2) + critique(2) = 5회에서 멈춘다. 품질이 끝까지 미달이어도 마지막 draft가 그대로 사용자에게 나간다 — 빈 응답 대신 "덜 좋은 동화"를 주는 정책.

STAGE ④

FINALIZE — 확정 노드 (LLM 호출 없음)

draft를 story로 옮겨 담는 단순 변환. 후처리 훅을 끼워넣기 위한 자리.

동작

def finalize(state):
    return {"story": StoryOut(
        title=state.get("draft_title", "오늘의 동화"),
        body=state.get("draft_body", ""),
    )}

✅ 정상

title/body가 채워진 StoryOut이 최종 상태에 들어가고 그래프 종료(END).

❌ 빈 body

모든 폴백을 거쳐도 body가 비어 있으면 service.generate_story 레이어에서 StoryParsingError로 변환 — 502 매핑.

요약: "미달" 상황별 처리

상황발생 단계처리최종 결과
LLM 응답 JSON 깨짐 plan raw 텍스트로 fallback해서 write에 그대로 전달 동화는 만들어짐 (품질은 critique이 검증)
LLM 응답 JSON 깨짐 write 정규식으로 title/body 추출 → 그것마저 실패 시 원문 전체를 body로 동화는 만들어짐
LLM 응답 JSON 깨짐 critique safety 0점 + passed=False → 재집필 트리거 최대 1회 재집필 후 그대로 finalize
safety 점수 누락 critique 0점 강제 — 평균이 4.0을 못 넘음 재집필 루프 진입
품질 평균 < 4.0 critique passed=False → write 재호출 (이번엔 critique 코멘트 첨부) 2차 통과 시 finalize, 또 미달이어도 한도 초과로 finalize

설계 의도 — 왜 이렇게 쪼갰나

1. 단일 책임 LLM은 더 잘 한다

한 번의 LLM 호출에 "기획 + 집필 + 검수"를 다 시키면 어느 하나가 약해진다. 분리하면 각 시스템 프롬프트가 짧고 명확해지고, 모델이 한 가지 역할에 집중한다.

2. 객관적 평가 가능

critique은 자기가 쓴 글이 아닌 독립된 컨텍스트의 draft를 본다 → self-evaluation의 bias를 줄인다. 4축 정량 점수 덕분에 통과/실패 분기가 결정론적이다.

3. 자율성 ↔ 안전장치의 이중 레이어

  • 프롬프트 레벨: "폭력/공포/훈계 금지" 같은 가이드
  • 코드 레벨: critique safety 0점 강제, 점수 클램핑, 파싱 fallback, MAX_REVISIONS 한도

LLM의 자유 출력을 신뢰하되, 핵심 안전 속성(점수 범위, safety 평가 누락 차단, 무한 루프 방지)은 코드가 보장한다.

4. 빈 응답보다는 덜 좋은 동화

모든 폴백·강제 finalize는 "사용자에게 빈 화면을 보여주지 않는다"는 정책이다. 품질이 미달이어도 가능한 한 동화를 돌려주고, 진짜 시스템 실패(LLM 호출 실패, body 추출 완전 실패)에서만 예외를 던진다.
