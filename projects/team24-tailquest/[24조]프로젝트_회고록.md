**팀**: 소태호(팀장), 원동규, 이강식, 임세현, 최준원
**기간**: 2026-05-03 ~ 2026-05-09
**문서 목적**: 발표용 회고. 기획 진화 → 구현 결정 → 디버깅 사건 → 데모 피드백 → 향후 방향.

## 0. 프로젝트 한 줄

> 사용자 답변을 LangGraph로 분석해 RAG·웹 검색·HITL을 거쳐 면접관 톤의 꼬리 질문을 생성하는 코칭 서비스.

**스택**: FastAPI + SQLAlchemy + Chroma · LangGraph + Solar Pro 3 · Next.js 14 + Tailwind · Docker + nginx + Tavily.

## 1. 기획서 진화 — v1 → v2 → v3 (구현 반영)

기획은 3번 갈렸다. 가장 많이 흔들린 결정은 **HITL의 의미와 범위**.

### "HITL"의 두 가지 뜻 — 본 프로젝트가 구분한 핵심 용어

| 의미 | 본 프로젝트 상태 |
|---|---|
| **(A) 사용자 답변이 다음 액션을 결정** — 답변 → 분석 → 다음 질문 / "잘 모르겠어요" → 토픽 전환 | ✅ **유지·강화** |
| **(B) Agent 출력에 대한 메타 피드백 HITL** — `interrupt_before`로 정지 → 사용자가 "regenerate / easier / harder / accept" 평가 → 그래프에 재주입 | ❌ **v3에서 제거** |

### 버전별 변경 요약

| 항목 | v1 (양식) | v2 (RAG 보강판) | v3 (구현 반영판) | 사유 |
|---|---|---|---|---|
| HITL(B) 메타 피드백 | — | 필수 | **제외** | 쉽다/어렵다 피드백을 정량 평가할 방법 없음 |
| 4-way 피드백 UI | — | 필수 | **제외** | 위 사유 + UX 번거로움 |
| 품질 평가 재생성 루프 | — | 필수 | **제외** | iteration_count 분기 자체 제거 |
| 검색 방식 | Dense | BM25+Dense Hybrid | **Dense 단일** | MVP 단순화, 한국어 정확도는 청크 임계값으로 보강 |
| 도메인 선택 | 다중 권장 | 다중 권장 | **단일 선택** | 다중 시 broad한 질문 발생 |
| 프론트엔드 | (미정) | Streamlit | **Next.js 14** | Airbnb 톤 viewport-bound + 향후 STT 확장 대비 |
| 상태 저장 | — | SqliteSaver | **MemorySaver + sessionStorage** | HITL(B) 제거로 resume 시나리오 소멸 |
| 자동 토픽 전환 | — | — | **신규** | 같은 도메인 3회 또는 uncertain/incorrect 시 |
| 새로고침 복원 | — | — | **신규** | sessionStorage `tq:chat_turns` 영구화 |
| 답변 길이 하한 | 50자 | 50자 | **1자** | "잘 모르겠어요" 같은 짧은 답변 처리 |
| LLM 실패 시 | mock fallback | mock fallback | **silent mock 폐기** | 진짜 에러 노출 + 답변 자동 롤백으로 동일 답변 재시도 가능 |

핵심: v2가 너무 야심차게 그렸던 그래프를 v3에서 **단순화**한 것이 가장 큰 의사결정.

## 2. LangGraph 노드 진화

```
[v2 계획]
START → analyzer ∥ term_extractor → knowledge_retriever
        → question_generator → evaluator
        → (실패 & iter<3) ── 재생성 ──┐
                                    │
        → (통과) → human_review (interrupt_before)
                       │
                       └─ feedback (regenerate/easier/harder/accept) → 재진입

[v3 실제 구현]
START → analyzer ∥ term_extractor → knowledge_retriever
        → question_generator (단일 probe)  ← drill-down 강제 + 답변 인용 강제
        → evaluator (점수만, 분기 미사용) → END

        + 클라이언트 자동 휴리스틱:
          - good + 같은 토픽 < 3회 → probe 푸시
          - good + 같은 토픽 ≥ 3회 → 새 seed
          - uncertain/incorrect    → 새 seed

[작업 후반 추가]
knowledge_retriever 안에서 deterministic Tavily fallback
   (Chroma chunks 0건 또는 max(score)<0.5일 때 자동 web 검색)

seed / question_generator 노드 안에서 LLM-driven tool calling
   structured_chat() ─→ tool_calling_chat() + structured_chat_after_tools()
   LLM이 web_search(query)를 자율 결정하여 시의성 정보를 주입
```

### 노드별 변화 요약

| 노드 | 처음 → 마지막 | 변경 이유 |
|---|---|---|
| `analyzer` | 약점 노트 | + answer_quality(good/uncertain/incorrect) + "잘 모르겠어요" 시 모범답안 + question_intent | "잘 모르겠어요"도 학습 기회로 만들기 |
| `term_extractor` | 변경 없음 | RAG 쿼리 강화용 |
| `knowledge_retriever` | Chroma 단일 → Chroma + 80자 미만 청크 폐기 + Tavily fallback | RAG 노이즈 제거 + 사용자 자료 빈약 시 웹 보강 |
| `question_generator` | 난이도별 3개 후보 → **단일 probe** + drill-down + 답변 인용 강제 → tool_calling으로 web_search 가능 | 후보 선택 단계가 번거롭고, 사용자 표현을 정조준해야 면접관답다는 판단 |
| `evaluator` | 점수 + 임계값 분기 + iteration_count | 점수만 출력, 분기 미사용 | 재생성이 품질 향상에 결정적이지 않았음 |
| ~~`human_review`~~ | interrupt_before + 4-way 피드백 → **노드 자체 제거** | 정량 평가 불가 |
| (신규) **클라이언트 자동 전환 휴리스틱** | — | 추가 | "사용자 의사 반영"의 의도는 살리되 그래프 밖으로 |

핵심: **workflow.py 토폴로지는 거의 안 변했고**, 모든 진화는 노드 내부 구현·상태 필드·외부 도구 통합으로 이루어짐.

### 상태(GraphState) 필드 추가 이력

| 필드 | 추가 시점 | 용도 |
|---|---|---|
| `retrieved_context: list[Chunk]` | RAG 추가 | knowledge_retriever 결과 누적 |
| `feedback: UserFeedback \| None` | HITL(B) | 도입 후 v3에서 사용 안 됨 |
| `feedback_count: int` | HITL(B) | 재시도와 분리된 별도 카운터 |
| `Chunk.source: "public"\|"user"\|"web"` | Tavily fallback | 인용 카드 분기 (web만 anchor) |
| `Chunk.url`, `Citation.url` | Tavily | 외부 링크 보존 |

## 3. 결정타였던 5가지 인프라 문제

### 3-1. 500 Internal Server Error — Next.js 14 rewrite의 30초 하드 타임아웃

**현상.** 답변 제출이 일정 비율로 500. backend 로그엔 traceback 0건. 프로세스는 16시간째 살아 있음.

**조사.** OOM·systemd 재시작·rate limit 모두 기각. 5번 호출 중 2번이 정확히 30.013초 / 30.017초에 실패하는 패턴 → frontend 로그에서 `Failed to proxy ... ECONNRESET` 발견.

**원인.** Next.js 14 `next.config.mjs` rewrite proxy의 **공식 문서에 없는 30초 하드 타임아웃**. LangGraph 응답이 17~29초 사이에서 변동하다 30초를 넘기면 Next가 upstream 소켓을 끊고 21바이트짜리 `Internal Server Error`를 반환. 그 사이 FastAPI는 끝까지 정상 처리.

**해결.** 운영 nginx에 `location /api/backend/`를 `location /` 앞에 두고 곧장 `127.0.0.1:8000`으로 우회. Next 리라이트 건너뜀. 5/5 외부 호출이 18~29초로 들쭉날쭉해도 모두 200.

**배운 점.** 로그가 답하지 않을 때는 **타이밍·재현 패턴**이 답한다. "정확히 30초"가 단서.

### 3-2. 채팅 히스토리 부재

**문제.** turn 데이터가 sessionStorage에만 — 탭 닫으면 사라짐. 좌측 사이드바·과거 세션 다시 보기·사용자별 격리 모두 부재.

**해결.**
- BE에 SQLite + 동기 SQLAlchemy 도입 (async 라우터에서는 `asyncio.to_thread`로 위임 — `aiosqlite` 추가 의존 없음).
- `sessions` 1 ↔ `turns` N. **`session_id`는 DB 식별자, `thread_id`는 LangGraph checkpointer 키** — 한 세션 안의 turn 마다 새 thread_id로 책임 분리.
- 신규 엔드포인트 `GET /sessions`, `GET /sessions/{id}`, `DELETE`, 나중 `PATCH`(rename) 추가.
- 프론트는 `/chat/[sessionId]` 라우트 + 좌측 280px 사이드바.

### 3-3. RAG 컨텍스트 부족

**1단계 (deterministic).** `knowledge_retriever_node` 끝 부분에 Tavily fallback 추가. Chroma chunks가 0건 또는 max(score)<0.5일 때 자동 web 검색. workflow.py 토폴로지는 무변경 — 같은 책임이라 노드 추가하지 않고 같은 노드 안에서 분기.

**2단계 (LLM-driven).** seed·question_generator 두 노드를 `structured_chat()` 단일 호출 → `tool_calling_chat()` + `structured_chat_after_tools()` 2단계 ReAct로 전환. LLM이 `web_search(query, max_results)`를 자율 결정.

### 3-4. 도커화 — 운영 무중단 컷오버

**제약.** 운영 EC2의 nginx (`/api/backend/` 직결 + Let's Encrypt)는 절대 손대지 않는다.

**설계.** BE/FE 컨테이너만 만들고 nginx는 호스트 그대로. 컨테이너 포트는 `127.0.0.1:8000`/`127.0.0.1:3000`로 묶어 외부 노출은 nginx만 담당.

**걸렸던 트랩 — Next.js 14 `rewrites()`는 build 시점에 baked.**
```dockerfile
ARG BACKEND_URL=http://backend:8000
ENV BACKEND_URL=${BACKEND_URL}
```
이걸 모르면 컨테이너 내부 frontend가 backend로 못 가서 ECONNREFUSED. **빌드 후 env는 의미 없다.**

### 3-5. Tool calling이 발동 안 함 — 가장 흥미로웠던 디버깅

`enable_llm_tool_calling=true`로 켜고 시의성 키워드(`Kubernetes 1.30 신규 기능`)로 호출했는데도 LLM이 자체 지식만으로 7.4초 만에 즉답. web_search 0건.

**원인 추적.**
1. `logger.info`가 stdout에 안 보임 → uvicorn logger propagation 문제 → `logger.warning`으로 격상.
2. 로그 보강 후: `tool_calling_chat: no tool_call fired — model answered directly.`
3. 즉, Solar Pro 3가 `tool_choice="auto"`를 무시.

**원인 분석.** SEED_GENERATOR_SYSTEM이 *"이 토픽으로 한 문장 만들어라"* 식으로 강하게 anchored. LLM이 도구 사용 가이드 없이 즉답을 선호.

**해결.** `tool_calling_chat` 안에서 system + user 양쪽에 자동으로 도구 힌트를 부착. system은 배경 지침, user는 즉시 컨텍스트.

**결과.**
```
tool_calling_chat: iter=0 fired 1 tool_call(s): ['web_search']
tool_calling_chat: iter=1 fired 1 tool_call(s): ['web_search']
```
응답 7.4초 → 20.7초. LLM이 검색 → 결과 → 추가 검색 → 최종 답변.

**배운 점.** "tool_choice='auto' 줬으니 부르겠지"는 거짓. **system prompt가 도구 사용을 명시적으로 권하지 않으면 zero-shot tool calling은 잘 안 일어난다.**

## 4. 데모에서 드러난 미해결 이슈 — 본질은 인프라가 아니라 LLM 비결정성

이 섹션이 회고의 핵심. 인프라는 잡았지만 **꼬리 질문 자체의 품질**이 면접관 수준엔 못 미친다.

### 4-1. 꼬리 질문 연관성

| 현상 | 사용자 데모 코멘트 |
|---|---|
| 답변·분석에서 동떨어진 질문 | "갑자기 딥다이브하는 느낌. 면접자가 언급하지 않은 것을 꼬리질문으로 물어봐서 의도와 달라지는 경우" |
| 분석 → 꼬리 질문 연결이 끊김 | "분석 결과 → 꼬리 질문이 이어지는 게 중요할 것 같음" |
| 면접관 수준 못 미침 | "질문의 퀄리티가 아직까지 면접관 수준은 아닌 것 같음" |

**현재 대응 (v3).** question_generator 프롬프트에 **drill-down 강제** + **답변 인용 강제** + **새 분야 도입 금지** + **출력 단일** 명시. analyzer notes의 critical/moderate 항목 한 개를 정조준.

**한계.**
- prompt에 예시를 넣었더니 **오버피팅** 발생 — 누가 답변을 하든 예시와 비슷한 질문이 나옴.
- LLM이 사용자 답변의 빈약한 부분을 표현해내는 게 생각보다 어려움.

### 4-2. HITL(B) 피드백 정량 평가 불가

| 시도 | 결과 |
|---|---|
| "쉽다/어렵다" 4-way 피드백 + 재생성 루프 | 새 질문이 정말 더 쉬운지/어려운지를 자동 판정할 방법 없음 → 노드 자체 제거 |

**향후 검토.** 새 질문 직후 **난이도 검증 노드**를 두어 약점 깊이·추상도 등을 평가하고 피드백 일치 여부를 채점. 통과 못하면 재생성. (회의 제안)

### 4-3. 다중 도메인 시 broad한 질문

| 현상 | 사용자 데모 코멘트 |
|---|---|
| 도메인을 다양하게 고려해서 질문하는 느낌 | (긍정) |
| broad한 질문이 나오는 경우 | "여러 도메인 선택했을 때 어떤 질문이 나와야 하는지에 대해 복잡해지는 경우" |

**v3 대응.** 단일 도메인 선택 + 자유 키워드 다중. 부분 적용.

**향후 검토.** 도메인 1개 끝까지 마친 후 자동 새 onboarding으로 **세션 분리**.

### 4-4. UI 불필요 컴포넌트

디자인 시스템(Airbnb 톤)을 가져다 쓰면서 임의로 들어간 컴포넌트들이 있음. 데모 후 사용자 자체 코멘트:

> "디자인 시스템의 컴포넌트가 임의로 들어감. 불필요하거나 사용하지 않는 것들은 정리해야 할 것 같음."

### 4-5. 분절된 대화 경험

> "사람이 한 번 말하고 AI가 답하고, 이게 분절되는 형태로 느껴질 수도 있을 것 같음."

turn-by-turn이 실제 면접감과 거리가 있음. STT를 검토했으나 **필수 기능 안정화가 우선**이라 후순위.

## 5. 컨셉 피보팅 검토 — "관련 질문 생성"으로 좁히기

회의에서 나온 대안:

> *"실제 면접 같은 환경을 포기하고, 연관된 질문을 생성해주는 컨셉으로 피보팅. 생각했던 것보다 범위를 한정지어서 하기. 지금은 어떤 면접 준비를 하던지 대응할 수 있게 하다보니까 잘 안 되는 것 같음."*

| 컨셉 | 형태 |
|---|---|
| (현재) 실제 면접 시뮬레이션 | 답변 → 분석 → 꼬리 질문 → 자동 진행. 면접관 톤. |
| (피보팅 후보) 관련 질문 생성기 | 특정 분야의 연관 질문을 다 제시 + 사용자 응답을 분석해주는 에이전트 |

**팀 의견.** 피보팅하더라도 지금까지 구현한 그래프·RAG·UI는 거의 그대로 변형 가능. 즉 작업이 무용해지지 않음. STT 같은 추가 기능은 컨셉 안정화 후.

| `aiosqlite` 비동기 드라이버 | ❌ 동기 + `to_thread` | 분당 수십 요청 미만, 의존성 1개 절약 |
| alembic 마이그레이션 | ❌ create_all + wipe-and-recreate | schema 변경 빈도 낮음 |
| 로그인 = 이메일/비번 + JWT | ❌ username 한 줄 | 데모, ChatGPT 식 가벼운 격리로 충분 |
| `tool_choice="required"` | ❌ `tool_choice="auto"` + 강한 prompt | 일반 개념엔 검색 안 부르도록 자율성 보존 |
| BM25+Dense Hybrid 검색 (v2 계획) | ❌ Dense 단일 + 청크 임계값 | MVP 범위 |
| 4-way 피드백 + 재생성 루프 (v2 계획) | ❌ 자동 진행 + 클라이언트 휴리스틱 | 정량 평가 불가 |

---

## 7. 못 보고 지나칠 뻔한 트랩 5선

1. **Next.js 14 rewrite proxy 30초 하드 타임아웃** — 문서 어디에도 없음. 정확히 30초 패턴이 단서.
2. **Next.js `rewrites()` destination은 build 시점에 baked** — 컨테이너에 env만 넣으면 안 통한다. ARG로 build에 박아야 함.
3. **SQLAlchemy `create_all`은 ALTER TABLE 못 한다** — `user_id` 컬럼 추가 시 옛 SQLite와 충돌. wipe + recreate 또는 alembic.
4. **Solar `tool_choice="auto"`는 system prompt가 강하면 무시된다** — system + user 양쪽에 도구 힌트가 박혀 있어야 ReAct 시작.
5. **Windows uvicorn `--reload`는 두 번 reload하면 죽는다** — `Finished server process` 후 재시작 안 됨. 로컬 dev에서는 `--reload` 빼고 수동 재시작 권장.

## 8. 회고 — 발표 마무리용 한 줄씩

- **로그가 답하지 않을 때는 타이밍·재현 패턴이 답한다.** 30초 타임아웃은 traceback이 아니라 stopwatch가 알려줬다.
- **노드를 늘리지 않고 책임을 늘리는 것이 더 안정적**이었다. Tavily fallback도 tool calling도 모두 기존 노드 안에서 구현 → trace는 한 줄로 보임 + 회귀 영역 최소.
- **제로샷 tool calling은 거짓말이다.** `tools=...`만 넣어도 안 부른다. system + user에 명시 가이드가 있어야 비로소 LLM이 도구를 쓴다.
- **무중단 컷오버는 인프라보다 결정에서 온다.** "운영 nginx는 손대지 않는다"는 한 줄 결정이 컷오버 다운타임을 5~15초로 줄였다.
- **하위 호환은 nullable로 시작해서 점진 강제로 끝낸다.** `user_id` 컬럼은 nullable로 추가, FE는 항상 동봉, 옛 row는 wipe로 정리 — 한 PR 안에서 마무리.
- **v2의 야심을 v3에서 깎은 것이 옳았다.** HITL(B)·재생성 루프·Hybrid 검색·다중 도메인을 모두 끄고 단순화했더니 그제야 핵심 — 답변과 꼬리 질문 사이의 연관성 — 에 집중할 수 있게 됨.
- **남은 어려움은 인프라가 아니라 LLM 비결정성에 있다.** 꼬리 질문 연관성·HITL 정량 평가 모두 prompt 디자인의 문제. 검증·평가 노드 재도입 + 다른 LLM 비교가 다음 단계.

## 부록 A. 회의 토론 발췌

> *Q. 질문 품질 평가나 피드백 구조가 없는데 일부러 뺀 건지 구현 전인지?*
> A. 구현해봤는데 잘 안돼서 일단 제거. HITL/워크플로우 구조 자체는 좋아 보였지만 **쉽다/어렵다 피드백을 넣었을 때 다음 질문이 정말 더 쉬운지/어려운지 정량 평가할 방법이 없어** 제거.

> *Q. 컨셉을 "관련된 질문 생성"으로 바꾼다면?*
> A. 지금까지 구현한 방식을 그 컨셉에 맞게 변형. STT는 필수 기능 미완성으로 후순위.

> *Q. 현재 어려운 점은?*
> A. (1) 꼬리 질문이 답변·분석과 동떨어지는 경우, (2) 사용자 피드백 반영 후 난이도 적합성 판단 어려움. **두 문제 모두 LLM 비결정성에 의존**.

> *Q. LLM API는?*
> A. Solar Pro 3. **다른 LLM API로 바꾸면 더 나아질지 검증 가치 있음**.

> *Q. 도메인/키워드 정책?*
> A. 다중 도메인 시 broad한 질문 발생 → **단일 도메인 + 세션 분리 모델 검토**.

> *데모 평가*: 완성도는 충분. **prompt 개선** + **검증·평가 노드 재도입** 등 agent 능력 발휘 부분이 추가되면 더 좋음.
