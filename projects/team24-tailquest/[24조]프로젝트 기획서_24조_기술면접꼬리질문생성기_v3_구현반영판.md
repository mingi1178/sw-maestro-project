# [24조] 프로젝트 기획서 (v3 — 구현 반영판)

**주제**: 기술 면접 꼬리 질문 생성기
**참여자**: 소태호(팀장), 원동규, 이강식, 임세현, 최준원
**최초 작성**: 2026-05-03 (RAG 보강판 v2) / **갱신**: 2026-05-07 (v3)
**문서 목적**: v2 대비 변형·축소·확장된 사항과 회의·리뷰에서 결정된 사유를 정리.

> **용어 정리 — "HITL" 두 가지 의미**
>
> 본 문서에서 "HITL 제거"라고 할 때는 항상 **(B)** 를 가리킨다.
>
> | 의미 | 본 프로젝트 상태 |
> |---|---|
> | **(A) 사용자 답변이 다음 액션을 결정** — 답변 → 분석 → 다음 질문 / "잘 모르겠어요" → 토픽 전환 | ✅ **유지·강화** |
> | **(B) Agent 출력에 대한 메타 피드백 HITL** — `interrupt_before` 로 그래프 정지 → 사용자가 "regenerate / easier / harder / accept" 평가 → 그래프에 재주입 | ❌ **제거** |

## 1. 문제 정의

**프로젝트 한 줄 정의**: LLM·RAG·LangGraph로 기술 면접 답변의 약점을 분석하고, 그 약점 한 곳을 정조준한 꼬리 질문을 한 개씩 자동으로 이어가는 면접 코칭 시스템.

**서비스 한 줄 정의**: 면접 답변과 본인 자료(md/pdf/GitHub)를 함께 입력하면 AI가 답변의 약점을 분석하고 그 약점에 정조준한 꼬리 질문을 자동으로 이어가며, 같은 토픽이 충분히 다뤄졌다고 판단되면 다른 토픽으로 자연스럽게 전환하는 맞춤형 면접 연습 코치.

**선정 배경 (v2 그대로)**: 기술 면접에서 첫 질문보다 꼬리 질문에서 탈락이 많고, 기존 도구는 정적 질문 목록에 그치며, 혼자 준비하는 취준생은 본인 답변 약점을 스스로 진단하기 어려움.

**해결하려는 문제**:
- **답변 깊이 부족 미인지**: 개념을 암기 수준으로만 이해해 "왜?", "어떻게?" 후속 질문에 대응 못함. 본인 답변이 피상적인지 스스로 판단 기준 없음.
- **핵심 용어 정확성 검증 부재**: 기술 용어를 정확히 사용했는지 검증할 기회 없음.
- **맥락 기반 꼬리 질문 부재**: 범용 챗봇은 답변 내용과 무관한 일반 질문만 생성.
- **사용자 맞춤화 부재** (RAG로 해결): 본인 정리 노트·관심 도메인 자료를 학습 루프에 직접 활용하기 어려움.

**대상 사용자**:
- IT/SW 직군 기술 면접을 준비하는 취준생·이직 준비자
- CS 기초(OS/네트워크/DB/자료구조) ~ 특정 기술 스택(Spring/React 등)까지 면접 대비
- 혼자 준비하거나 스터디에서 꼬리 질문 질을 높이고 싶은 사람
- 본인 정리 노트·GitHub 면접 자료를 학습 루프에 직접 활용하고 싶은 사람

**핵심 가치**: "내 자료, 내 답변에 맞춘 꼬리 질문" — 정적 질문 목록이 아니라 사용자가 실제로 한 답변을 분석하고 본인 자료의 출제 패턴을 참고하여 면접관 관점에서 파고들 포인트를 찾아냄.

## 2. Agent 설계

### 타깃 사용자 페르소나

| 항목 | 내용 |
|---|---|
| 연령대 | 20대 중반 ~ 30대 초반 |
| 직업 | CS 전공 대학생, 부트캠프 수료생, 주니어 개발자(이직 준비) |
| 특징 | 개념은 학습했으나 면접에서 깊이 있게 설명하기 어려워함. 혼자 연습 시 피드백 부재. GitHub·블로그 면접 자료를 모으지만 활용 방법이 막연함. |

### 노드 구성 (구현 반영)

| 노드 | 상태 | 역할 / 변경 |
|---|---|---|
| `analyzer_node` | ✅ | answer_quality(good/uncertain/incorrect) 분류 + 약점 노트 + 잘 모르겠어요 시 모범답안 생성 + 질문 의도 정리 |
| `term_extractor_node` | ✅ | 답변 내 영문/한글 기술 용어 추출 (RAG 쿼리 강화용) |
| `knowledge_retriever_node` | ✅ | Chroma 벡터 검색. **80자 미만 청크는 noise로 폐기** |
| `question_generator_node` | ✅ | **단일 probe 모델로 변경**. v2: 난이도별 3개 후보 → v3: 분석 notes 중 critical/moderate 항목 한 개를 정조준한 단일 질문 |
| `evaluator_node` | ⚠️ | 점수만 출력. **재생성 분기·임계값 분기 제거** (분기 단순화) |
| ~~`human_review_node`~~ | ❌ | **제거** — Agent 출력에 대한 메타 피드백 HITL(B). 사용자 피드백(쉽게/어렵게)을 정량 평가할 방법이 없어 그래프에서 분리 |
| (신규) **클라이언트 자동 전환 휴리스틱** | ✅ | 같은 도메인 follow-up 3회 후 또는 answer_quality가 uncertain/incorrect → 자동 새 seed |

> **요약**: 6노드 + 조건부 엣지 2개 → **5노드 선형 그래프**. (B) HITL의 의도 중 "사용자 의사 반영"은 클라이언트 측 자동 전환으로 옮김 — (A) 흐름은 오히려 강해짐.

### Agent 성격 및 톤앤매너

- 엄격하지만 건설적인 면접관 — 약점을 정확히 짚되 보완 방향 함께 제시
- 존댓말, 회화체 면접관 톤 ("혹시 ~ 들어보셨어요?", "방금 ~ 라고 하셨는데 ~ 는 어떻게 되나요?")
- 불필요한 칭찬 없이 핵심만 전달
- 자료 인용 시 출처(file_name + heading) 표시로 신뢰성 확보

### Agent의 자율성 범위

| 구분 | 자율 수행 | 사용자 승인 필요 | v2 → v3 |
|---|---|---|---|
| 답변 분석 및 약점 식별 | ✅ | | 그대로 |
| 핵심 용어 추출 | ✅ | | 그대로 |
| 지식 베이스 검색 및 컨텍스트 구성 | ✅ | | 그대로 |
| 꼬리 질문 초안 생성 | ✅ | | 그대로 |
| 꼬리 질문 품질 평가 | ✅ (점수만) | | 분기 미사용 |
| **최종 꼬리 질문 확정 및 제시** | ✅ **자동 진행** | ❌ | v2: 사용자 검토 → v3: 자동 push |
| 사용자 자료 업로드/삭제 | | ✅ | 그대로 |

### 통합 워크플로우

사용자 액션 → BE LangGraph 실행 → 클라이언트 라우팅:

```
[onboarding] 트랙 → 도메인(단일) + 키워드 → [선택: 자료]
        │
        ▼
[chat: seed 자동 생성]
        │
        ▼  ┌──── 사용자 답변 또는 [잘 모르겠어요] ◄─┐
        │  │                                       │
        ▼  ▼                                       │
   ┌─────────────────────────────────────┐         │
   │  LangGraph (선형)                    │         │
   │   analyzer ∥ term_extractor          │         │
   │     → knowledge_retriever            │         │
   │     → question_generator (단일 probe)│         │
   │     → evaluator → END                │         │
   └─────────────────────────────────────┘         │
        │                                           │
        ▼                                           │
   분석 카드 + 인용 출처 + 다음 질문 자동 push      │
        │                                           │
        ▼  클라이언트 라우팅:                       │
        │   - good + 같은 토픽 < 3회 → probe 푸시 ──┤
        │   - good + 같은 토픽 ≥ 3회 → 새 seed ────┤
        │   - uncertain/incorrect → 새 seed ───────┘
        │
        ▼
[브라우저 종료까지 반복; sessionStorage에 turns 영구화]
```

> **v2 대비 변경**: HITL(B) 단계(사용자가 후속 질문을 검토·재생성·수용 결정) 제거. 자동 진행 + 토픽 자동 전환으로 대체. (A) 흐름은 답변·"잘 모르겠어요" 클릭이 직접 다음 질문을 결정하므로 더 직접적.

### 시나리오 (대표 1개)

**김개발 (자료 미등록)**: /onboarding → "운영체제" 단일 도메인 → seed 자동 생성("컨텍스트 스위칭이...") → 답변 → analyzer가 약점("PCB 설명 부재") → question_generator가 그 약점 정조준 probe("방금 컨텍스트 스위칭 비용 짚어주셨는데, PCB가 무엇이고 비용에 직접 영향을 주는지 설명해주실 수 있을까요?") → 자동 진행. 같은 토픽 3회 후 또는 잘 모르겠어요 시 → 다른 토픽으로 자동 전환.

자료 등록 시나리오: /materials 에서 GitHub 레포/파일 업로드 → BE가 청킹·HTML strip·임베딩 → seed 단계에서 사용자 자료 청크 마이닝 → citation card와 함께 출제. 청크가 모두 80자 미만이면 자동으로 정적 풀 폴백.

### RAG 핵심 설계

**(1) 자료 인입**: 파일(.md/.pdf), GitHub 레포 URL, Raw URL 3가지 경로.

**(2) 청킹 + HTML strip (v2 대비 추가)**:
- 마크다운 헤딩 단위 청킹 (그대로)
- HTML 태그·엔티티 정규식 strip (`<details>`/`<summary>`/`<p>` 등) — GitHub 면접 자료의 collapsible Q&A가 임베딩 노이즈를 만들기 때문
- HTML strip 후 **80자 미만 청크는 폐기** — 빈약한 자료가 LLM 답변 품질을 낮춤

**(3) 검색 단순화**:
- v2의 BM25+Dense Hybrid → Dense 단일
- Top-K = 5
- 검색 결과에 retrieve-time HTML strip + 80자 임계 한 번 더 (방어층)

**(4) Question Generator 프롬프트 핵심**:
- **drill-down 강제**: notes 중 critical/moderate 항목 한 개 정조준
- **답변 인용 강제**: 사용자 표현·용어를 그대로 인용해서 질문
- **새 분야 도입 금지**: 분야 전환은 클라이언트 휴리스틱이 담당
- **출력 단일**: `follow_ups` 배열에 정확히 한 개

| 계층 | 위치 | 영구화 |

| 항목 | v2 | v3 |
|---|---|---|
| 답변 길이 하한 | 50자 | **1자** ("잘 모르겠어요" 같은 짧은 답변 처리) |

### 성공 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 꼬리 질문 관련성 | 생성된 꼬리 질문의 80% 이상이 원래 답변과 직접 관련 | 사용자 피드백(유용함/불필요) 비율 |
| 워크플로우 완주율 | 입력부터 최종 꼬리 질문 확인까지 90% 이상 정상 완료 | 시스템 로그 |
| RAG 인용률 | 사용자 자료 등록 세션의 70% 이상에서 검색 컨텍스트가 최종 질문에 반영 | Question Generator 출력의 citations 채워진 비율 |
| RAG 만족도 향상 | 자료 등록 사용자가 미등록 사용자보다 "유용함" 피드백 비율 +15%p 이상 | A/B 비교 |
| 인입 성공률 | 업로드/URL 인입 작업의 95% 이상 정상 완료 | Ingestion 파이프라인 로그 |

> **v2 대비**: HITL 동작 지표(재생성 후 만족도 향상)는 HITL(B) 제거로 측정 대상 자체가 사라져 폐기.

## 6. 알려진 이슈 & 개선 방향

회의 결론: 핵심 어려움 두 가지(질문 연관성·HITL 정량 평가)는 **구현 난이도보다는 LLM 비결정성에 의존**. 단기 개선 경로는 (1) prompt 디자인 강화, (2) 검증·평가 노드 재도입 검토, (3) 다른 LLM 비교.

| 이슈 | 현상 | 단기 대응 |
|---|---|---|
| 꼬리 질문 연관성 | 답변·분석에서 동떨어지는 경우 | prompt에 좋은/나쁜 예시 페어 다양화, analyzer notes 정조준 항목을 schema로 구조화 |
| HITL(B) 피드백 정량 평가 어려움 | 쉽게/어렵게 피드백 후 새 질문이 정말 그런지 자동 판정 불가 | 난이도 검증 노드 재도입 시 evaluator 분기 다시 활성화 |
| 다중 도메인 broad 질문 | 도메인 간 맥락 섞임 | 단일 도메인 + 세션 분리 (현재 단일 선택 부분 적용) |
| 분절된 대화 경험 | turn-by-turn 구조가 실제 면접감과 거리 | 음성 인터페이스(Phase A: STT) 또는 면접관식 보조 발화 추가 |

## 부록. 회의·리뷰 토론 발췌

> *Q. 질문 품질 평가나 그 기반 피드백 구조가 없는데 일부러 뺀 건지 구현 전인지?*
> A. 구현해봤는데 잘 안돼서 일단 제거. HITL이나 워크플로우 구조 자체는 좋아 보였지만 **쉽다/어렵다 피드백을 넣었을 때 다음 질문이 정말 더 쉬운지/어려운지를 정량적으로 평가할 방법이 없어** 제거.

> *Q. 컨셉을 "관련된 질문 생성"으로 바꾼다면?*
> A. 지금까지 구현한 방식을 그 컨셉에 맞게 변형하는 정도. STT는 필수 기능이 안 되는 상황이라 도입 후순위.

> *Q. 현재 어려운 점은?*
> A. (1) 꼬리 질문이 답변·분석과 다소 동떨어지는 경우, (2) 사용자 피드백 반영 후 난이도 적합성 판단 어려움. **두 문제 모두 LLM 비결정성에 의존하는 부분이 본질**. 예시 다양화 + 검증 노드 추가 방식으로 해결 가능 여부 검토.

> *Q. LLM API는?*
> A. Solar Pro 3 사용 중. **다른 LLM API로 바꾸면 더 나아질지 검증 가치 있음**.

> *Q. 도메인/키워드 정책?*
> A. 다중 도메인 시 broad한 질문 발생 → **단일 도메인 + 세션 분리 모델 검토**. 키워드는 분야와 가까운 역할.

> *데모 평가*: 완성도는 충분. **prompt 개선** + **검증·평가 노드 재도입** 등 agent 능력 발휘 부분이 추가되면 더 좋음.
