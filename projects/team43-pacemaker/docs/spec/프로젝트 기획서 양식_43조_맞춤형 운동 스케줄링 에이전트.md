## **1\. 문제 정의 및 프로젝트 개요**

| 프로젝트 한 줄 정의(프로젝트 전체를 한 문장으로 요약) \[필수\] 사용자의 캘린더 일정과 건강/운동 데이터를 종합 분석하여 실행 가능한 최적의 맞춤형 운동 스케줄을 자동 생성하는 에이전트 구현. 서비스 한 줄 정의(사용자 관점에서 이 서비스가 무엇인지 한 문장으로 설명) \[필수\] 바쁜 일정과 컨디션에 맞춰, 이번 주 언제 어떤 운동을 해야 할지 알아서 계획해주는 서비스. 서비스 선정 배경(왜 이 주제를 선택했는가) \[권장\] 현대인 다수는 운동의 필요성에 공감하지만, 실제 실행 단계에서 두 가지 장벽에 부딪힌다. 첫째, 매주 변동되는 일정 속에서 운동 시간을 확보하는 일. 둘째, 자신의 체력 수준에 맞는 루틴을 직접 설계하는 일이다. 이 두 가지가 누적되며 발생하는 인지적 부담은 운동 지속의 핵심 장애물로 작용한다. 본 서비스는 AI가 일정 조율과 루틴 설계를 대신 수행함으로써 이 부담을 제거하고, 사용자의 실행력을 높이고자 한다. 해결하려는 문제(핵심으로 생각하는 문제, 문제라고 생각한 이유를 함께 기재) \[필수\] 핵심 문제: 운동을 지속하려는 사용자가 매주 부딪히는 두 가지 장벽 — (1) 가변적인 일정 속에서 운동 가능한 시간대를 식별하는 일, (2) 당일의 건강 상태와 피로도를 고려해 적정 강도의 운동을 설계하는 일. 문제로 본 이유: 두 작업 모두 단일 결정이 아니라 매번 새로운 변수에 따라 다시 수행해야 하는 반복적 의사결정이다. 이러한 반복은 누적된 인지 부담을 만들고, 결국 "오늘은 그냥 쉬자"는 회피 행동으로 이어진다. 즉 본질적인 문제는 운동 자체에 대한 동기 부족이 아니라, 운동을 둘러싼 계획 비용이라고 판단했다. 대상 사용자: (누가 이 Agent를 사용하는가?) \[필수\] 규칙적인 운동을 원하지만 야근, 약속 등 변동성 높은 스케줄을 가지고 있어 고정된 시간에 운동하기 힘든 2030 직장인 및 대학생. 핵심 가치: (이 프로젝트가 사용자에게 주는 가장 큰 이점) \[필수\] 사용자의 개입 없이 다중 데이터(일정+건강+운동기록)를 연동하여 매주 현실적으로 실행 가능한 구체적인 운동 시간표와 맞춤형 루틴을 제공하는 압도적인 편의성. |

## **2\. 사용자 및 Agent 설계**

| 타깃 사용자 페르소나(ex, 연령대, 직업, 특징 등) \[권장\] 20대 후반 백엔드 개발자. 잦은 야근으로 평일 일정이 불규칙함. 스마트워치로 수면 시간은 측정하지만 이를 기반으로 내일 무슨 운동을 할지 결정하지는 못하는 상태. Agent의 역할(최대한 구체적으로 기재) \[필수\] 사용자의 이번 주 캘린더 일정을 스크래핑하여 가용 시간 추출. 사용자의 최근 건강 데이터(수면량, 활동량) 및 과거 운동 성취도 데이터 조회. 사용자에게 직관적인 선택을 돕기위해 해당 시점의 근육 부하량을 이미지화. 위 데이터를 종합하여 사용자에게 운동 종목, 강도, 일정 추천 Agent의 성격 및 톤앤매너 \[권장\] 전문적이고 격려하는 퍼스널 트레이너 (“목요일은 야근 일정이 있으시네요. 무리하지 마시고 퇴근 후 15분 가벼운 스트레칭 루틴은 어떠세요?”) Agent의 자율성 범위: (어디까지 스스로 결정하고 실행할 것인가? 예: 정보 검색 후 요약까지만 수행) \[필수\] 사용자의 캘린더 및 건강 데이터를 읽고 분석하여 스케줄을 생성하는 것까지 자율적으로 수행하며, 최종 캘린더 일정 등록은 사용자의 최종 승인 필요. |

## **3\. 핵심 기능 및 사용자 흐름**

| 주요 사용자 시나리오 \[필수\] 사용자는 바쁜 일상속에서 여러 운동을 취미로 즐기고 있으나 출근과 다양한 개인 일정으로 인해 언제 어떤 운동을 할지 계획하는데에 불편함을 느낌. 사용자는 이 서비스에 접속하여 자신의 개인 일정에 더해 운동 계획을 확인할 수 있고 현재의 근육 부위별 피로도를 이미지로 직관적으로 확인함. 사용자는 채팅창을 통해 운동을 계획하는데에 있어서 고려되었으면 하는 자신의 요구사항을 입력 (예) 앞으로 7일 동안의 운동 일정 추천해줘. 4일에는 약속이 잡힐 가능성이 있으니 추천에서 제외하고, 3일정도 가는것을 목표로 하고 있는데 내 일정에 맞춰서 추천해줘. 오랜만에 러닝도 포함되면 좋겠어. Agent가 일정, 요청사항을 기반으로 운동 계획 생성 해당 일정에는 운동 부위별 부하도와 개인의 수면시간등의 건강 데이터가 반영됨. (예) 월요일 22:00 홈트(스쿼트), 수요일 19:00 헬스(등) , 금요일 17:00 러닝(10km) 핵심 기능 정의 \[필수\] 도구 호출 기능: 가상의 일정 데이터, 건강 데이터, 과거 운동 기록 DB를 조회하는 Tool 구현. 스케줄링 추론 엔진: 수집된 데이터를 바탕으로 LLM이 상황에 맞는 운동 스케줄을 도출하는 로직. 대화형 인터페이스: 제안된 일정에 대해 “화요일은 너무 피곤할 것 같아” 라고 피드백하면 스케줄을 재조정하는 기능. 이미지 생성: 확정된 일정대로 운동이 진행될 경우 해당 일자의 근육 피로도 이미지 생성 기능 워크플로우 사용자 관점 워크플로우 \[필수\] 서비스 접속 → 운동 계획 생성 요청 → Agent 가 분석 및 제안 완료 → 제안된 일정 확인 및 수정 요청 → 최종 승인 및 캘린더 등록. 시스템 관점 워크플로우 \[권장\] 사용자 Prompt 입력 → Agentic 로직 실행 → 캘린더, 기존 운동 데이터 조회 → 결과 종합하여 최종 답변 포맷팅 → UI 출력 → 수정 요청 → Prompt에 맞춰 필요한 일정만 변경 |

## **4\. 기술 구현 설계**

| 기술 스택 \[필수\] 언어: Python LLM & Agent 프레임워크: OpenAI GPT-4o & LangGraph UI: Streamlit 시스템 아키텍처 \[권장\] 로컬 PC 환경 내에서 구동. Streamlit 프론트엔드가 LangGraph 기반의 Agent 와 통신하며, Agent 는 로컬에 구축된 가상의 사용자 데이터(일정, 건강 상태 (애플워치를 가지고 단축어 앱 통해서 바로 공유 가능)) JSON 파일을 Tool을 이용해 읽어와 분석함. 프롬프트 설계 전략 \[권장\] ReAct 패턴 적용: 시스템 프롬프트에 “반드시 1)일정 확인, 2)건강 상태 확인, 3)운동 기록 확인의 3단계를 거친 후 스케줄을 짤 것”을 명시하여 환각을 방지하고 툴 사용을 강제함. 데이터 활용 및 기억 관리 \[권장\] 대화 메모리를 활용하여, 한 번 제안한 스케줄 내역을 기억하고 사용자의 추가 수정 요구사항을 반영할 수 있도록 설계. 제약사항 및 예외 처리 \[필수\] 제약사항: 보안 및 인증 이슈로 실제 Google Calendar나 Apple Health 연동여부 \- 출시전에는 애플헬스 불가능하다 판단 별도의 데이터 사용. 예외 처리: 일주일 내내 빈 시간이 전혀 없는 경우, 10분 단위의 홈트나 계단 오르기 등 대체하여 제안하는 엣지 케이스 로직 추가. |

## **5\. 성과 평가 및 실행 계획**

| 성공 지표(KPI) \[권장\] Agent 가 생성한 운동 스케줄이 사용자의 기존 캘린더 일정과 시간적 충돌이 발생하는 비율 1% 미만 달성. 근육 피로도에 맞춰 적절한 일정이 제안되는 비율 체크 → 해당 일자까지 근육 피로도가 높음인데 해당 근육을 주로 사용되는 운동이 추천되는 경우 실패. MVP 범위 \[필수\] Agent가 사용자의 일정·수면·운동 기록 등 건강 데이터를 읽고 추론하여, 이번 주 맞춤형 운동 스케줄표를 화면에 출력하고, 각 일자별 근육 피로도를 이미지로 생성해 함께 제공한다. 단계별 개발 로드맵 \[필수\] 1주차 (5/1 \~ 5/3) — 학습 및 설계 5/1(금) \~ 5/2(토): 팀원 각자 LangGraph 학습 및 본 기획 구현 방안 정리. 5/3(일): 팀 미팅 — LangGraph 학습 내용 공유, 각자 정리한 구현 방안 발표, 아키텍처·역할 분담·기술 스택 최종 합의. 2주차 (5/4 \~ 5/10) — 개발 5/4(월) \~ 5/5(화): 개발 환경 세팅, 가상 데이터(JSON) 구축 — 캘린더 일정, 건강(수면·활동량), 과거 운동 기록. 5/6(수) \~ 5/7(목): LangGraph 기반 Agent 코어 구현 — Tool(데이터 조회) 정의, ReAct 추론 노드, 스케줄 도출 로직. 5/8(금): Streamlit UI 구현 및 Agent 연동, 대화형 피드백(스케줄 재조정) 기능 연결. 5/9(토): 통합 테스트, 엣지 케이스 보완(빈 시간 없는 경우 10분 대체 루틴 등), 메모리/멀티턴 검증. 5/10(일): 개발 완료, 코드 동결 및 데모 시나리오 동작 확인. 3주차 (5/11 \~ 5/15) — 발표 준비 5/11(월) \~ 5/12(화): 발표 자료 작성, 데모 시나리오 스크립트화, 화면 녹화/스크린샷 준비. 5/13(수) \~ 5/14(목): 내부 리허설, 피드백 반영, 발표자료·데모 환경 최종 점검. 5/15(금): 최종 발표. 기대 효과 \[권장\] 운동 계획 수립에 드는 시간과 고민을 없애주어 운동 진입 장벽을 낮추고, 내 컨디션에 꼭 맞는 현실적인 목표 설정을 통해 건강 관리의 지속성을 획기적으로 높일 수 있음. |
