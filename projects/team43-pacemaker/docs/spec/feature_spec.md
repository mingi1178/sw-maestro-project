## 1. 한 줄 요약

바쁜 일정과 컨디션을 종합해서, 이번 주에 언제 어떤 운동을 할지 자동으로 짜주는 챗봇 형태의 웹 앱 (Flutter Web 프론트 + FastAPI/LangGraph 백엔드).

### 페르소나

20대 후반 백엔드 개발자. 야근이 잦아 평일 일정이 매주 바뀐다. 현실에서는 스마트워치로 수면을 측정하는 사람이지만, "그래서 내일 뭐 할까"는 직접 못 정한다. 헬스, 러닝, 홈트를 번갈아 한다.

> **MVP 범위 주의**: 위 페르소나는 사용자상을 그리기 위한 배경 묘사입니다. 이 MVP는 실제 Apple Health 또는 스마트워치 API에 연동되지 않으며, 수면/활동량 데이터는 `data/health.json`에 미리 넣어둔 가짜 값을 사용합니다 (기획서 4번, 루트 `CLAUDE.md` 10번 명시).

## 3. 핵심 기능 7개

각 기능마다 다음을 명시합니다.

- **사용자 행동**: 사용자가 무엇을 하는가
- **시스템 응답**: 시스템이 무엇을 보여주는가
- **수용 기준 (AC)**: 이 기능이 "끝났다"고 말하기 위한 통과 조건
- **담당 슬라이스**: A~E (역할 분담, `../planning/plan.md`). 화면은 A/B, 데이터는 D/E, 추론은 C.

### F1. 이번 주 일정 보기

- **사용자 행동**: 앱에 진입한다.
- **시스템 응답**: 좌측 상단에 이번 주(월~일) 캘린더 일정이 카드 형태로 보인다.
- **수용 기준**:
  1. 카드에는 일자, 시작/종료 시각, 일정 제목이 표시된다.
  2. `is_busy=true`와 `false`가 시각적으로 구분된다 (색상 또는 아이콘).
  3. 일정이 0건이면 "이번 주 일정이 비어 있어요" 안내가 뜬다.
  4. 데이터 소스는 Supabase `calendar_events` 테이블 (Flutter가 직접 조회. 실제 Google Calendar 미연동).
- **담당 슬라이스**: A (FE 카드) + D/E (데이터·Tool)

### F2. 이번 주 컨디션 보기

- **사용자 행동**: 앱에 진입한다.
- **시스템 응답**: 좌측 중간에 최근 7일 평균 수면시간, 평균 활동시간, 평균 안정시 심박이 카드로 보인다.
- **수용 기준**:
  1. 평균 수면 5시간 미만이면 "피로 누적" 배지가 카드에 표시된다.
  2. 활동시간이 7일 평균 30분 미만이면 "활동량 부족" 배지가 표시된다.
  3. 데이터 소스는 Supabase `health_snapshots` 테이블 (Flutter가 직접 조회. 실제 Apple Health 미연동).
- **담당 슬라이스**: A (FE 카드) + D/E (데이터·Tool)

### F3. 최근 운동 이력 보기

- **사용자 행동**: 앱에 진입한다.
- **시스템 응답**: 좌측 하단에 최근 7일 운동 기록이 시간 역순으로 리스트 표시.
- **수용 기준**:
  1. 각 항목에 일자, 운동명, 시간(분), 강도(1~5), 대상 부위가 표시된다.
  2. 빈 기록은 "최근 운동 이력이 없어요"로 처리.
  3. 데이터 소스는 Supabase `workout_records` 테이블 (Flutter가 직접 조회).
- **담당 슬라이스**: A (FE 카드) + D/E (데이터·Tool)

### F4. 운동 스케줄 자동 추천 (★ 메인 기능)

- **사용자 행동**: 채팅창에 요청을 입력한다. 예: "앞으로 7일 운동 스케줄 추천해줘. 3일 정도 가는 게 목표고, 러닝도 포함해줘."
- **시스템 응답**: Agent가 일정/컨디션/운동 이력을 모두 조회한 뒤, 일자별 추천 슬롯을 채팅창에 출력한다.
- **수용 기준**:
  1. 응답에는 입력에서 명시한 일자 범위만큼 일별 슬롯이 생성된다.
  2. 추천된 슬롯이 캘린더의 `is_busy=true` 일정과 시간이 겹치지 않는다.
  3. 피로도가 "높음"인 부위에 같은 부위를 쓰는 운동을 추천하지 않는다.
  4. 빈 시간이 0인 날은 10분 단위 대체 루틴(홈트, 계단)을 제안한다.
  5. 각 슬롯에 일자, 시작 시각, 운동명, 강도(1~5), 대상 부위가 모두 포함된다.
  6. Agent는 응답 전 반드시 `get_calendar`, `get_health`, `get_workouts` 3개를 차례로 호출한다 (ReAct 강제).
- **담당 슬라이스**: C (Agent 추론/그래프) + B (FE SSE 스트림 표시)

### F5. 부위별 피로도 시각화

- **사용자 행동**: F4 결과를 받는다 (별도 행동 없음).
- **시스템 응답**: 우측에 인체 실루엣 이미지가 표시되고, 부위별 누적 피로도가 색상으로 구분된다.
- **수용 기준**:
  1. 부위 7종(가슴, 등, 하체, 어깨, 코어, 이두, 삼두) 모두 색이 매핑된다.
  2. 피로도 0~5가 색의 진하기로 표현된다 (0=초록, 3=노랑, 5=빨강).
  3. F4의 추천 운동이 바뀌면 이미지도 같이 바뀐다 (F4와 동기화).
  4. 같은 입력 state에 대해 항상 같은 PNG가 나온다 (캐시).
- **담당 슬라이스**: A (Flutter 레이더 차트 위젯). 색·매핑은 C가 `MuscleFatigueState`로 전달.

### F6. 멀티턴 재조정

- **사용자 행동**: F4 응답을 받은 뒤 채팅창에 수정 요청을 입력한다. 예: "화요일은 너무 피곤할 것 같아 빼줘."
- **시스템 응답**: Agent가 이전 제안을 기억한 채로 해당 일자만 수정한 새 제안을 돌려준다.
- **수용 기준**:
  1. 명시된 일자만 변경되고 나머지 일자는 그대로 유지된다.
  2. 같은 세션 내에서 thread_id가 유지되어 이전 추천 컨텍스트가 보존된다.
  3. 사용자가 "왜 그렇게 바꿨는지" 물으면 Agent가 변경 사유를 한 문장으로 답한다.
  4. 재조정 후 F5 이미지도 같이 갱신된다.
- **담당 슬라이스**: C (메모리/재조정 프롬프트) + B (멀티턴 입력창)

### F7. 최종 승인

- **사용자 행동**: 추천이 마음에 들면 "캘린더에 등록" 버튼을 누른다.
- **시스템 응답**: 추천 슬롯들이 `POST /data/calendar`로 1건씩 추가되고 일정 카드가 갱신된다 ("등록 완료" 토스트).
- **수용 기준**:
  1. 추천이 없는 상태에서는 버튼이 비활성화 또는 숨김 처리된다.
  2. 클릭 시 Flutter가 Supabase `calendar_events` 테이블에 직접 슬롯을 추가한다 (`create_calendar_event` 패턴).
  3. 등록 후 좌측 일정 카드(F1)가 새 일정을 보여준다 (refetch).
  4. 채팅 이력은 유지된다.
- **담당 슬라이스**: B (FE 등록 버튼) + D (`create_calendar_event` Tool — F7 클릭 시 호출)

## 4. 기능과 코드 모듈 매핑

| F2 | `data/health.json` | `tools.get_health` ↔ `GET /data/health` | (호출만) | `frontend/lib/cards/health_card.dart` |
| F3 | `data/workouts.json` | `tools.get_workouts` ↔ `GET /data/workouts` | (호출만) | `frontend/lib/cards/workouts_card.dart` |
| F4 | 위 3종 | 위 3종 모두 | `agent/nodes.py::think, compose_schedule` → `proposal` 청크 | `frontend/lib/chat/` (B) |
| F5 | F4 산출 `MuscleFatigueState` | (없음) | (없음) | `frontend/lib/cards/fatigue_radar.dart` |
| F6 | F4 산출 + 사용자 피드백 | 위 3종 (재조회 가능) | `agent/nodes.py::refine` + `memory/` 체크포인터 | `frontend/lib/chat/` (B) |
| F7 | F4 산출 | `tools.create_calendar_event` ↔ `POST /data/calendar` | (없음) | `frontend/lib/chat/` (등록 버튼) |

## 5. KPI 시나리오 (5/9 통과 목표)

기획서 5번 그대로 옮긴 5개. 통합 테스트 통과 기준입니다.

1. **일정 충돌률 1% 미만** — 동일 입력으로 10회 추천 생성 시 `is_busy=true` 일정과 시간 충돌 0회.
2. **피로도 회피** — 피로도 "높음"(4 이상) 부위에 같은 부위 운동을 추천하는 사례 0회.
3. **빈 시간 대체** — 빈 시간 0인 주에 대해 10분 단위 홈트/계단 등 대체 루틴이 제안된다.
4. **멀티턴 재조정** — "화요일은 피곤할 것 같아" 입력 시 화요일 슬롯만 변경된다.
5. **이미지 일관성** — 추천된 부위와 F5 이미지의 색상 변화가 일치한다.

각 시나리오는 `tests/test_kpi.py`에 `@pytest.mark.kpi` 마커로 작성됩니다 (`tests/CLAUDE.md`).
